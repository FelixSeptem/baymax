## Context

Baymax 当前在模型、MCP、工具、编排与诊断层已经形成稳定主线，但面向外部集成者的“如何扩展适配器”仍缺统一样板与迁移地图。现有文档虽覆盖运行时配置和模块边界，但接入实践分散，容易造成接口用法不一致与错误语义偏移。

根据当前路线，A21 应聚焦 DX 收敛，不引入平台化能力，也不改 runtime 核心执行语义。

## Goals / Non-Goals

**Goals:**
- 提供文档主导的外部适配样板，并附最小代码模板。
- 按优先级交付接入样板：`MCP > Model > Tool`。
- 提供能力域 + 代码片段双维迁移映射，覆盖常见错误与替代写法。
- 将文档与索引纳入现有质量门禁，避免后续漂移。

**Non-Goals:**
- 不交付完整生产级外部 adapter 框架。
- 不新增 provider 协议实现或新传输协议。
- 不修改运行时契约字段与默认行为。

## Decisions

### 1) 输出形态采用“文档主导 + 最小代码模板”
- 方案：主交付为文档与模板骨架，不追求可直接部署。
- 原因：最大化可读性与可迁移性，同时控制实现范围。
- 备选：直接实现完整 adapter 工程。拒绝原因：范围过大且与 A21 目标不匹配。

### 2) 模板优先级固定为 MCP > Model > Tool
- 方案：先覆盖外部调用接入面最广的 MCP，再覆盖 Model，最后 Tool。
- 原因：与当前多代理主链路外扩需求一致。
- 备选：三类并行同等深度。拒绝原因：产出分散且难保证质量一致。

### 3) 迁移映射按“能力域 + 典型代码片段”双维组织
- 方案：每个能力域提供旧写法、推荐写法、替代路径、常见错误。
- 原因：兼顾架构理解与落地复制。
- 备选：仅提供字段级映射表。拒绝原因：缺少可执行接入上下文。

### 4) 兼容语义统一为 additive + nullable + default + fail-fast
- 方案：在迁移文档中明确此四项边界，并给出 parser/调用方建议。
- 原因：与现有 run summary / diagnostics 契约口径一致。
- 备选：按模块分别定义。拒绝原因：增加认知负担和漂移风险。

### 5) 门禁复用现有 docs consistency 与 contributioncheck
- 方案：不新增独立复杂 gate，扩展既有检查索引与文档一致性。
- 原因：保持治理成本可控且符合当前工作流。
- 备选：新增专门文档 gate。拒绝原因：收益低于维护成本。

## Risks / Trade-offs

- [Risk] 模板过于抽象，集成者仍难上手  
  → Mitigation: 每类模板附带“最小可运行片段 + 常见错误对照”。

- [Risk] 文档更新后无法持续保持一致  
  → Mitigation: 将样板/映射索引接入 docs consistency 与 contributioncheck。

- [Risk] 与 A20 示例内容重叠  
  → Mitigation: A20 保持“端到端演示”，A21 聚焦“外部接入模板与迁移映射”，边界分离。

## Migration Plan

1. 新增外部适配样板文档与模板目录结构。
2. 先补 MCP 样板，再补 Model 与 Tool 样板。
3. 建立双维迁移映射与常见错误章节。
4. 更新 README、roadmap、mainline index 导航。
5. 扩展 docs consistency/contributioncheck 追踪规则并验证。

回滚策略：
- 如发现模板表达不稳定，可先回滚模板链接与索引映射，保留既有文档主路径；
- 回滚不影响 runtime 行为与已归档能力。
