## Context

现有运行时诊断链路同时存在两类写入路径：
1) 业务组件（`core/runner`、`skill/loader`）直接写入诊断存储；
2) 事件录制器（`observability/event/runtime_recorder`）基于事件再次写入诊断存储。

该结构在并发、重试、事件重放场景下会导致重复记录，进而污染 run/skill 统计口径。与此同时，诊断契约（字段、状态枚举、错误语义）在多个入口分散维护，回归成本高。项目已有并发优化与异步通信能力，但质量门禁侧需要把并发安全验证固化为默认基线。

## Goals / Non-Goals

**Goals:**
- 将 run/skill 诊断写入收敛为单一写入通道，消除同语义双写。
- 为诊断写入建立幂等键与去重策略，在并发、重试、重放场景保持口径稳定。
- 统一诊断契约（schema、状态枚举、错误语义）并补齐契约测试。
- 将并发安全检查（含 race）纳入默认质量门禁，确保不可绕过。

**Non-Goals:**
- 不重构对外诊断 API 的整体形态与查询入口。
- 不引入新的 CLI 诊断命令或额外运维面板。
- 不扩展到性能基准框架重建，仅保持现有相对百分比门禁语义。

## Decisions

### 1) 采用 single-writer 策略，禁止业务直写与事件录制并存落库
- 决策：保留一个“规范写入入口”（由实现阶段选定为事件侧或业务侧之一），另一侧仅保留语义转换/转发，不允许落库。
- 理由：从架构上消除重复样本的根因，比后置清洗更可靠。
- 备选：双通路都保留并依赖查询层去重。
  - 放弃原因：查询层去重成本高、易漏场景、对延迟和复杂度不利。

### 2) 幂等键进入诊断写入模型，并在存储层强制去重
- 决策：定义稳定幂等键生成规则（覆盖 run 与 skill 记录），写入时执行存在性检查/覆盖策略。
- 理由：并发与重试是常态，幂等必须由写入层承担，不能依赖调用方“尽量不重复”。
- 备选：仅靠调用方携带 request-id。
  - 放弃原因：调用方分散，缺少全局约束，长期不稳定。

### 3) 契约集中化：统一状态枚举与错误分类，增加 contract tests
- 决策：在诊断领域模型中收敛状态与错误码枚举，run/skill 共享核心字段，保留子域扩展字段。
- 理由：语义一致性可测试，避免不同入口各自解释。
- 备选：按模块各自维护映射。
  - 放弃原因：维护成本与语义漂移风险高。

### 4) 并发安全作为强制质量门禁
- 决策：默认质量流必须包含 `go test -race ./...` 与关键并发测试集；失败即阻断。
- 理由：与并发优化目标一致，保证新增并发改动不会回归。
- 备选：仅在发布前或夜间任务跑 race。
  - 放弃原因：反馈周期过长，问题发现滞后。

## Risks / Trade-offs

- [Risk] 收敛单写入后，历史依赖另一通路的埋点可能缺失 → Mitigation: 在迁移期提供对照日志与双读校验，不做双写。
- [Risk] 幂等键规则不当可能误合并合法记录 → Mitigation: 为 run/skill 分别定义最小唯一字段集并补充边界测试。
- [Risk] race 检查增加 CI 耗时 → Mitigation: 分层执行（快速必跑 + 全量并发集），但 merge gate 仍保持强制。

## Migration Plan

1. 明确并切换到唯一写入入口，移除另一入口的落库动作。
2. 引入幂等键模型并完成写入层去重改造。
3. 统一诊断 schema/枚举/错误语义，更新 run/skill 生产端映射。
4. 增加契约测试与并发测试，接入默认质量门禁。
5. 更新 README/docs 与 change artifacts，说明口径与迁移注意事项。
