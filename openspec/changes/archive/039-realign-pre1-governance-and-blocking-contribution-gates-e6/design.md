## Context

仓库已落地一版开源治理基线，但当前文档默认了“稳定期开源项目”的外部承诺（兼容性承诺与安全响应时限）。在 `pre-1.x` 个人项目阶段，这类承诺与实际维护能力不匹配，容易造成对外预期偏差。与此同时，现有 Issue/PR 模板主要依赖人工遵循，缺少可执行的阻断级约束。

本提案属于治理与流程收敛，不触及 runtime 语义或 API 行为。

## Goals / Non-Goals

**Goals:**
- 将治理口径收敛为 `pre-1.x` 轻承诺策略：保留版本表达，去除兼容与 SLA 承诺。
- 统一安全入口为邮箱私报通道，保留可执行的最小披露流程。
- 将 Issue/PR 模板关键字段升级为必填，并由 CI 阻断执行。
- 保持中文优先模板体验，同时允许英文提交不被流程拒绝。

**Non-Goals:**
- 不新增或修改 runtime、context assembler、model adapter 逻辑。
- 不引入漏洞平台、自动化 SLA 仪表盘或响应指标承诺。
- 不扩展为多分支长期维护策略（仅维护最近 minor）。

## Decisions

### Decision 1: 版本策略采用“SemVer 表达 + pre-1.x 非承诺”
- 方案：继续使用 `MAJOR.MINOR.PATCH` 版本号表达，但明确 `1.0.0` 前不提供兼容性承诺。
- 原因：兼顾版本可读性与当前维护现实，避免过度承诺。
- 备选：维持现有兼容承诺文档；否决原因是与项目阶段不匹配。

### Decision 2: 安全通道固定为邮箱私报，移除 SLA
- 方案：`SECURITY.md` 明确邮箱报告渠道，保留流程步骤，移除响应时限目标。
- 原因：个人项目更适合低运营负担的单通道受理方式。
- 备选：继续 GitHub Security Advisory + SLA；否决原因是流程管理成本更高且非当前必要。

### Decision 3: 贡献模板采用“结构必填 + 语言宽容”
- 方案：模板字段中文优先命名与说明，允许英文内容；关键段落/勾选项必须存在。
- 原因：保证本地协作效率，同时不排斥英文贡献者。
- 备选：仅文档提示不强制；否决原因是执行一致性不足。

### Decision 4: 阻断门禁由仓库脚本 + CI required check 落地
- 方案：新增或扩展模板校验脚本，CI 中执行并对缺失必填项直接失败。
- 原因：治理约束必须可执行，才能稳定影响行为。
- 备选：仅依赖 GitHub 模板 required frontmatter；否决原因是对 PR body 的阻断能力有限、可绕过。

## Risks / Trade-offs

- [Risk] 移除兼容性与 SLA 承诺后，外部用户对升级风险判断成本上升
  -> Mitigation: 要求变更在 `CHANGELOG.md` 中明确 breaking 影响与迁移建议（信息透明但不作兼容承诺）。

- [Risk] 阻断级校验可能提高首次贡献门槛
  -> Mitigation: 模板仅保留最小必填项，并给出示例占位文本。

- [Risk] 纯脚本校验可能与 GitHub UI 细节耦合
  -> Mitigation: 校验规则基于稳定标题标记与勾选语义，避免依赖复杂 DOM 或 API 行为。

## Migration Plan

1. 更新治理文档（版本策略、安全策略、贡献指南）并统一引用入口。
2. 调整 Issue/PR 模板为中文优先、关键项必填结构。
3. 实现模板完整性校验脚本并接入 CI。
4. 将校验 job 设为分支保护 required check（仓库设置层）。
5. 以一次 PR 演练校验“缺字段阻断、字段完整放行”两条路径。

回滚策略：
- 若阻断规则过严导致贡献流程受阻，可先回退脚本或在 CI 暂时降级该检查；
- 文档口径回滚与代码路径解耦，不影响 runtime 功能。

## Open Questions
