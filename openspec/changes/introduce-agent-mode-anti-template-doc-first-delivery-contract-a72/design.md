## Context

`examples/agent-modes` 在 a62/a71 后形成了完整目录、README、矩阵与门禁接线，但当前代码仍存在明显同构模板形态：模式之间主要差异集中在常量、marker 与字符串拼接，真实模式语义执行深度不足。该状态导致两个直接问题：

1. 任务勾选与代码真实度脱节，出现“完成状态与实现质量不一致”。
2. 示例难以作为迁移参考，生产接入价值有限。

本提案目标是把“反模板”从口头约束升级为可验证契约，且强制执行“文档先行 -> 按文档实现 -> 门禁阻断回流”的交付顺序。

约束边界：
- 保持 `library-first`，不引入平台化控制面。
- 不重写既有 domain contract 语义，仅对示例交付与门禁治理做增强。
- shell/PowerShell gate 语义保持 parity。

## Goals / Non-Goals

**Goals:**
- 建立模式级文档基线，先明确每个模式的语义锚点、真实运行路径、验收证据与失败回滚。
- 按文档定义逐模式替换实现，禁止统一模板骨架仅参数化复用。
- 新增反模板阻断门禁，确保后续修改无法回退到模板实现。
- 把任务勾选口径收敛为“四证据齐全”（代码/测试/文档/门禁）。

**Non-Goals:**
- 不新增平台能力（控制平面、托管执行面、外部运维面板）。
- 不在本提案扩展新的业务 contract 范畴。
- 不要求所有模式接入全部 runtime 域，仅要求符合各模式职责域。

## Decisions

### Decision 1: 文档先行是强制前置阶段
- 方案：先完成 `MATRIX.md` 与模式 README 的语义基线，再允许进入代码替换阶段。
- 具体要求：每个模式先固定以下文档字段后，才可勾选实现任务：
  - `semantic anchor`
  - `real runtime path evidence`
  - `expected verification markers`
  - `failure / rollback notes`
- 原因：避免再次出现先套模板代码、后补文档导致语义漂移。
- 备选：代码与文档并行推进。
- 取舍：前期推进稍慢，但一致性与可审查性显著提高。

### Decision 2: 反模板判定采用“双引擎”策略
- 方案：反模板门禁由静态结构判定 + 运行证据判定组成。
- 静态结构判定：
  - 检测跨模式高同构骨架（归一化后指纹重复度超阈值）。
  - 检测 wrapper-only 入口（仅调用统一执行器、缺少模式自有业务逻辑）。
- 运行证据判定：
  - 必须命中模式声明域中的真实行为分支（非纯打印/拼接）。
  - `minimal` 与 `production-ish` 必须在行为路径上分叉，而非 marker 增量。
- 原因：单纯静态或单纯运行判定都容易漏报/误报。
- 备选：仅靠人工 review 或 smoke 输出关键字。
- 取舍：门禁复杂度提高，但能形成稳定阻断能力。

### Decision 3: 模式语义实现必须“模式内自有”
- 方案：每个模式在自身目录维护语义执行单元，禁止把业务语义集中到跨模式共享执行引擎。
- 共享代码仅允许用于非语义公共能力（输出格式、通用断言、工具函数）。
- 原因：共享语义引擎会天然鼓励“参数化模板”，与真实示例目标冲突。
- 备选：允许共享语义引擎，要求补更多 marker。
- 取舍：增加维护成本，但确保模式差异真实可读。

### Decision 4: 任务勾选必须绑定“四证据”
- 方案：每个模式任务必须同时满足以下条件才可勾选：
  - 代码证据：模式内语义执行已落地。
  - 测试证据：至少包含正向 + 退化/失败路径验证。
  - 文档证据：README 与矩阵已同步。
  - 门禁证据：anti-template + smoke + readme-sync 通过。
- 原因：避免“文档完成即勾选”或“代码占位即勾选”。
- 备选：保持当前宽松勾选。
- 取舍：短期勾选速度下降，长期质量显著上升。

### Decision 5: 以分批替换控制风险
- 方案：按 P0/P1/P2 分批推进，每批都要求文档与门禁闭环后再进入下一批。
- 原因：28 模式一次性替换回归风险高。
- 备选：全量一次替换。
- 取舍：周期更可控，失败可局部回滚。

## Risks / Trade-offs

- [Risk] 反模板规则过严导致误报。  
  -> Mitigation: 引入模式白名单例外文件并要求在提案中显式说明原因，默认最小化例外。

- [Risk] 文档先行增加前置成本。  
  -> Mitigation: 提供统一文档骨架与自动校验，减少重复编写成本。

- [Risk] 模式内自有逻辑导致代码重复。  
  -> Mitigation: 允许复用非语义工具层，禁止复用业务语义层。

- [Risk] 分批推进期间存在中间态。  
  -> Mitigation: 每批独立可验收、可回滚，不跨批勾选“完成”状态。

## Migration Plan

1. 建立 A72 文档基线：
   - 更新 `examples/agent-modes/MATRIX.md` 的模式语义字段与状态列。
   - 补齐所有模式 README 的必需章节与模式特定语义说明。
2. 落地 A72 门禁：
   - 新增 `check-agent-mode-anti-template-contract.*`。
   - 新增 `check-agent-mode-doc-first-delivery-contract.*`。
   - 接入 `check-quality-gate.*` 阻断流程。
3. 分批替换代码（P0 -> P1 -> P2）：
   - 每批先文档确认、再代码替换、再测试与门禁收口。
4. 收口与审计：
   - 执行全量 smoke 与质量门禁。
   - 产出 A72 验收报告并准备归档。

回滚策略：
- 若某批引入回归，回滚该批模式目录与对应文档/门禁改动，不影响前序批次稳定状态。
