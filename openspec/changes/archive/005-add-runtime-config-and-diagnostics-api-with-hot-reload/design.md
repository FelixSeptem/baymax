## Context

现有运行时已经具备并发控制、异步通信和 MCP 可靠性 profile，但缺少统一配置入口与在线动态调整机制。参数分散在初始化路径中，环境差异需要改代码或重启进程，且缺乏稳定的库级诊断查询接口。为降低运维与调优成本，本设计引入基于 Viper 的配置中心能力、API-only 诊断接口与热更新机制，并与现有 MCP runtime profile 行为对齐。

约束：
- 语言与运行时：Go，面向多 goroutine 并发执行模型。
- 配置优先级：`env > file > default`。
- 配置格式：YAML。
- 诊断交互：仅库 API，不新增 CLI。
- 错误策略：配置错误和关键初始化错误 fail-fast。

## Goals / Non-Goals

**Goals:**
- 提供统一运行时配置加载器，支持默认值、YAML 文件、环境变量覆盖和严格校验。
- 提供热更新能力：监听文件变更并原子替换生效配置，失败回滚到旧配置。
- 提供诊断查询 API：最近运行摘要、最近 MCP 调用摘要、生效配置快照（脱敏）。
- 让 `mcp/http` 与 `mcp/stdio` 共享同一配置源和诊断采样语义，保持一致性。
- 为并发读写路径建立线程安全基线（锁或原子指针），并补齐并发测试。

**Non-Goals:**
- 不提供 CLI 诊断命令。
- 不引入远程配置中心（如 etcd/consul）。
- 不改变既有业务语义（除配置来源与可观测性增强外）。

## Decisions

1. 使用 `github.com/spf13/viper` 作为配置加载核心
- 方案：通过 Viper 处理 YAML 读取、环境变量绑定、文件监听回调。
- 原因：社区成熟、Go 生态广泛使用、升级迁移成本低。
- 备选：自研解析+watch。放弃原因：重复造轮子，边界条件复杂。

2. 配置对象采用“不可变快照 + 原子切换”
- 方案：解析与校验后生成新配置快照，通过 `atomic.Value`（或等价手段）一次性替换。
- 原因：读路径无锁或低锁开销，避免半更新状态。
- 备选：全局可变结构体+RWMutex。放弃原因：误用风险高，更新期间更易出现读取不一致。

3. 热更新失败不污染当前运行态
- 方案：文件变更触发后执行 parse->validate->build snapshot，全流程成功后才 swap；任一步失败即返回错误并保持旧配置。
- 原因：保证可用性与可回退性。
- 备选：先写入后修复。放弃原因：会暴露中间不合法状态。

4. 诊断能力仅提供库接口
- 方案：在 `mcp/runtime` 暴露 `GetRecentRuns`、`GetRecentCalls`、`GetEffectiveConfig`（命名可按代码风格调整）等 API。
- 原因：最小化接口面，避免 CLI 兼容包袱。
- 备选：额外 CLI 子命令。放弃原因：超出范围，增加发布与维护成本。

5. 保持 fail-fast 语义
- 方案：初始化阶段若配置缺失/非法，立即返回错误；运行期热更新若失败，立刻记录错误并拒绝该次更新。
- 原因：行为可预测，避免隐性降级。
- 备选：自动降级静默修复。放弃原因：会掩盖配置问题。

## Risks / Trade-offs

- [Risk] 环境变量映射不一致导致覆盖失效 -> Mitigation: 明确前缀与键映射规则，提供映射表文档和单元测试。
- [Risk] 热更新频繁触发导致抖动 -> Mitigation: 引入去抖窗口与重复内容短路（哈希/版本比对）。
- [Risk] 诊断缓存过大影响内存 -> Mitigation: 固定环形缓冲区上限并提供可配置保留条数。
- [Risk] 并发读写下出现竞态 -> Mitigation: `go test` 并在 CI 开启 race 检查（支持环境下）。

## Migration Plan

1. 增加新配置模块（schema、默认值、viper loader、validator、watcher）。
2. 在运行时入口接入配置加载，并将现有 profile 参数迁移为从统一配置快照读取。
3. 增加诊断 API 并替换旧的分散式诊断数据采集点。
4. 打开热更新并补齐集成测试（成功更新、非法更新回滚、并发读取一致性）。
5. 更新 README 与 docs，包括配置字段、优先级、环境变量映射和热更新行为。

## Open Questions

- 当前已确认不提供 CLI；若后续需要运维入口，可在后续提案追加 thin wrapper，不纳入本次范围。
- `GetEffectiveConfig` 的脱敏字段清单可先按常见凭证字段实现，并在文档标注 TODO 扩展策略。
