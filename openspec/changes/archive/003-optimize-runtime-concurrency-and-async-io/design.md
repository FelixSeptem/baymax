## Context

现有 Baymax v1 已支持多轮 loop、工具调度和 MCP 双传输，但并发执行路径仍以功能正确为主，缺少系统化并发策略（统一限流、背压、异步通信、取消收敛）与量化性能门禁。与此同时，团队计划参考 PocketFlow tutorials 扩容 examples，如果底层并发/异步机制和诊断能力不足，示例会出现“可演示但不可稳定复现”的问题。

## Goals / Non-Goals

**Goals:**
- 定义并实现运行时并发控制层，统一并发度、队列、背压、超时和取消传播。
- 定义异步通信管线，支持工具/MCP 调用的异步任务编排和事件回传。
- 建立并发安全基线，强制纳入 `go test -race ./...` 与 goroutine 泄漏检查。
- 建立性能回归门禁，以相对提升百分比作为性能优化和回归评估标准。
- 以 roadmap 节奏分批扩容 examples，并在每个示例保留 TODO 扩展位。

**Non-Goals:**
- 不引入分布式执行或跨进程编排（R4 方向）。
- 不在本提案内完成全量高级示例一次性交付。
- 不改动核心模型协议语义（仅增强并发/异步运行机制与质量门禁）。

## Decisions

### Decision 1: 先底座、后示例扩容
- Choice: R1/R2 优先运行时并发与异步机制，R2 后段和 R3 扩容 examples。
- Rationale: 保证示例和文档建立在稳定能力之上。
- Alternatives:
  - 先扩容示例：短期可见，但技术债和行为不稳定风险更高。

### Decision 2: 性能目标采用相对提升百分比
- Choice: 以“相对当前基线提升 X%”评估吞吐与延迟改进。
- Rationale: 避免受机器/环境绝对值波动影响，便于长期追踪趋势。
- Alternatives:
  - 绝对阈值：不同环境下可比性较弱，维护成本更高。

### Decision 3: 并发安全作为强制质量门禁
- Choice: 必须通过 race、泄漏检查、取消风暴测试，未通过不得合入。
- Rationale: 并发正确性是性能优化前提。
- Alternatives:
  - 非强制建议：执行不稳定，难形成组织基线。

### Decision 4: 异步通信采用显式通道与关联事件语义
- Choice: 使用 channel/context 构建异步任务通讯，事件中固定关联字段和任务状态。
- Rationale: 降低跨模块耦合，提高诊断与回放能力。
- Alternatives:
  - 隐式回调链：可实现但可观测性和可维护性较差。

### Decision 5: 示例采用分批模板化扩容并保留 TODO
- Choice: R2 批次 A（基础/中阶），R3 批次 B（并发/异步高级），每个示例保留 TODO。
- Rationale: 让外部接入路径清晰，后续优化可持续演进。
- Alternatives:
  - 一次性全量示例：交付风险和维护成本过高。

## Risks / Trade-offs

- [并发策略复杂度上升] → 先收敛核心策略集合，避免过度可配置化。
- [异步链路引入更多时序 bug] → 强制 race + 泄漏 + 取消风暴测试，配合 golden 场景。
- [性能优化与功能迭代冲突] → 设置独立 benchmark 基线与回归阈值，分支内强制对比。
- [示例扩容拖累主线交付] → 分批推进并在 tasks 中显式设置 TODO 管理。

## Migration Plan

1. 新增并发/异步配置对象与默认值，不破坏现有默认行为。
2. 在 runner/tool/mcp 路径逐步接入并发控制和异步管线。
3. 增加指标与事件字段，确保向后兼容消费方。
4. 接入并发安全门禁与性能回归门禁。
5. 交付 R2 批次 A 示例并附 TODO，后续在 R3 扩展批次 B。

## Open Questions

- 相对提升百分比的初始阈值按功能域分层还是统一阈值。
- goroutine 泄漏检查是否引入统一测试辅助库或自研轻量工具。
- example TODO 模板是否包含“性能预期/已知限制/扩展方向”固定段落。
